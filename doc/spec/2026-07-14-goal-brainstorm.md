# Skill `goal-brainstorm`

## Назначение

`goal-brainstorm` превращает исходное намерение пользователя в согласованный набор документов-владельцев и короткую исполняемую цель Codex. Skill поддерживает создание новой спецификации, изменение существующих инструкций, design и specification документов, создание `doc/goal/**` и опциональную активацию persistent goal после отдельного подтверждения пользователя.

Skill временно распространяется в plugin `workflow-container-tools`. Он остаётся generic и не получает workflow-container-specific семантику, чтобы позднее его можно было перенести без изменения публичного контракта.

## Scope

Skill должен:

- исследовать реальное состояние проекта до проектирования;
- читать применимые `AGENTS.md` и релевантные design/specification документы;
- задавать ноль или больше вопросов по одному, только пока остаются существенные неоднозначности, и не спрашивать факты, которые можно достоверно получить из проекта;
- выявлять противоречия, скрытые допущения, границы scope и критерии готовности;
- проектировать проверку каждого изменяемого наблюдаемого поведения без написания тестового кода;
- предлагать альтернативы только для реальных design decisions;
- изменять правильные owner-документы после утверждения соответствующих решений;
- создавать отдельную implementation specification только когда она действительно нужна;
- всегда создавать один goal-файл для общей реализации;
- опционально активировать persistent goal только после отдельного явного подтверждения.

Skill не должен:

- начинать реализацию во время brainstorm;
- создавать отдельный implementation plan;
- автоматически создавать commit или выполнять push;
- изменять или очищать уже активную persistent goal;
- дублировать требования между instructions, design, spec и goal;
- содержать какую-либо логику или терминологию удаляемого Superpowers workflow.

## Владельцы требований

- `AGENTS.md` владеет постоянными инструкциями и инженерными ограничениями своего path scope.
- `doc/design.md` и релевантные `doc/design/**.md` владеют стабильной архитектурой и проектными контрактами.
- `doc/spec/**` владеет task-specific требованиями реализации, которые не принадлежат естественно существующим owner-документам.
- `doc/goal/**` владеет краткой исполняемой целью и точными ссылками на утверждённые источники.

Goal не становится новым владельцем design или implementation requirements. Spec не копирует постоянные инструкции или уже определённую архитектуру.

## Выбор изменяемых документов

Перед проектированием skill определяет все затронутые owner-документы и выбирает один режим.

### Direct Owner Update

Отдельная implementation specification не создаётся, когда требования представляют собой небольшие изменения существующих instructions, design или spec либо когда даже значительное изменение полностью и естественно принадлежит одному или двум существующим owner-файлам. Skill изменяет эти файлы непосредственно и создаёт goal со ссылками на точные изменённые разделы.

### Dedicated Implementation Specification

Новый файл `doc/spec/**` создаётся, когда задача имеет самостоятельное большое ТЗ, пересекает несколько владельцев, требует единого task-level контракта, содержит migration/rollout или совместные acceptance criteria нескольких компонентов либо загрязнила бы стабильные owner-документы временными деталями реализации.

Dedicated spec может сопровождаться изменениями существующих instructions, design и других spec. Она содержит только объединяющие task-specific требования.

## Имена файлов

Новые документы используют дату создания и semantic name:

```text
doc/spec/YYYY-MM-DD-<semantic-name>.md
doc/goal/YYYY-MM-DD-<semantic-name>.md
```

Путь существующей spec после создания не меняется. Новый goal получает текущую дату и может ссылаться на более старую spec. При совпадении имени в тот же день существующий файл обновляется только для той же задачи; другая задача получает более точный semantic name вместо числового суффикса.

Для multi-repository задачи dedicated spec и один goal создаются в координирующем repository. Ссылки на контракты внутри него остаются root-relative, а ссылки на другие repositories используют абсолютные пути. Если владелец координации не следует однозначно из задачи и структуры проектов, skill спрашивает пользователя.

Существующий goal-файл обновляется только для продолжения той же неактивной цели. Новая цель или продолжение уже завершённой цели получает новый goal-файл с текущей датой. До изменения существующего goal-файла skill проверяет persistent goal state и не меняет файл, на который ссылается незавершённая goal, без отдельного явного решения пользователя.

## Lifecycle

1. Определить repository root, применимые `AGENTS.md`, релевантные design/spec/goal документы, затронутый код и тесты.
2. Определить владельцев требований и до редактирования показать пользователю список создаваемых и изменяемых документов с объяснением каждого выбора.
3. Задавать вопросы по одному только пока остаются существенные неоднозначности, проверяя текущие факты по проекту и не создавая искусственных design alternatives.
4. Представить решение и его verification obligations самостоятельными смысловыми разделами и получить подтверждение каждого раздела.
5. После полного согласования изменить утверждённые owner-документы и при необходимости создать dated implementation spec.
6. Выполнить semantic self-review и исправить найденные противоречия, дубли, неясные owner boundaries, незакрытые вопросы и непроверяемые требования.
7. После разрешения всех owner-конфликтов создать один dated goal-файл, выполнить его semantic reread вместе с source contracts и показать итоговые документы и diff.
8. Отдельно спросить об активации показанной версии цели. До подтверждения не начинать implementation work.
9. При подтверждении повторно проверить persistent goal state и применить явную матрицу переходов:
   - если status tool недоступен, не вызывать creation tool, сообщить, что состояние не проверено, и вывести точную команду `/goal` с условием запуска только при отсутствии незавершённой цели;
   - если status показывает незавершённую goal, сообщить о ней, не выводить команду активации и после её разрешения проверить status снова;
   - если status подтверждает отсутствие незавершённой goal и creation tool доступен, активировать цель из goal-файла;
   - если status подтверждает отсутствие незавершённой goal, но creation tool недоступен, сообщить только о невозможности автоматической активации и вывести точную команду `/goal`.

После активации brainstorm lifecycle завершён. Дальнейшая реализация выполняется обычным persistent goal lifecycle.

## Контракт спецификации

Спецификация использует структуру, соответствующую задаче, а не обязательный универсальный набор заголовков. В ней должны присутствовать все применимые смысловые элементы:

- требуемый результат и решаемая проблема;
- подтверждённое текущее состояние;
- scope и non-goals;
- утверждённые решения и причины выбора;
- целевое поведение;
- публичные интерфейсы, модели и владельцы данных;
- состояния и переходы для stateful поведения;
- обработка ошибок и восстановление;
- migration и compatibility, когда они нужны;
- изменения по проектам и owner-компонентам;
- verification obligations и проверяемые acceptance criteria для каждого изменяемого наблюдаемого поведения.

В утверждённой spec не остаются альтернативные решения, открытые вопросы, `TODO`, placeholders или промежуточные bridge-состояния. Spec описывает конечное состояние, но не предписывает подробную последовательность implementation steps.

## Контракт Goal

Goal должен быть существенно короче 4000 символов и содержать:

```markdown
# <Название результата>

## Outcome

<Конкретное конечное состояние.>

## Source Contracts

- `<path>`: `<точный раздел или роль документа>`

## Constraints

<Только task-specific границы, которые не выражены ссылками.>

## Verification

<Проверяемое определение готовности или точные ссылки на его владельцев.>
```

Goal описывает outcome, constraints и verification, предоставляет точный file context и оставляет Codex свободу построить и корректировать собственный план. Он не копирует source contracts, не содержит устаревающий предполагаемый список implementation files и требует завершить весь scope, выполнить применимые проверки и проверить итоговый diff на дублирование, лишние wrappers, размазанную ответственность и противоречия.

Persistent objective указывает на goal-файл и требует считать его completion contract. Публичные рекомендации Codex применяются без неподтверждённых model-specific правил для `gpt-5.6-sol`.

Goal-контракт основан на официальных документах OpenAI `Long-running work` (`https://learn.chatgpt.com/docs/long-running-work.md`) и `Prompting` (`https://learn.chatgpt.com/docs/prompting.md`): outcome описывает конечный результат, constraints задают только существенные границы, verification делает завершение проверяемым, а подробный контекст передаётся через файлы вместо перегруженного objective.

## Конфликты и безопасность

Если намерение пользователя противоречит действующему owner-документу, skill показывает конфликт и предлагает изменить настоящего владельца вместо маскировки противоречия новой spec. После утверждения затронутые документы должны описывать один целевой контракт.

Если owner-конфликт остаётся нерешённым, skill останавливается без противоречащих правок и goal-файла и сообщает точное требуемое решение. Обязательное создание одного goal относится только к успешно завершённому brainstorm lifecycle.

Skill сохраняет несвязанные пользовательские изменения, не перезаписывает неизвестный документ до определения его роли и не создаёт несколько goal-файлов для одной реализации.

Перед изменением существующего goal-файла и непосредственно перед активацией skill вызывает `get_goal` или эквивалентную доступную проверку. Если status tool недоступен, существующий goal-файл не изменяется и replacement-файл не создаётся до доступной проверки состояния. При незавершённой цели новая goal остаётся неактивной до отдельного решения пользователя; команда активации не выводится. Skill не очищает, не заменяет и не завершает существующую цель самостоятельно.

## Структура Skill

```text
plugins/workflow-container-tools/skills/goal-brainstorm/
  SKILL.md
  references/
    specification-contract.md
```

`SKILL.md` владеет interaction workflow, capability matrix и terminal behavior. `references/specification-contract.md` владеет выбором owner-документов, форматами spec/goal, verification design и semantic self-review. Скрипты не добавляются: решения зависят от семантики проекта, а не от механического преобразования.

Skill должен поддерживать explicit invocation. Plugin metadata, `workflow-container-tools/AGENTS.md` и skill `workflow-container-developer` обновляются так, чтобы новый skill был видимым, но не становился workflow-container-specific owner.

## Миграция

После добавления и проверки `goal-brainstorm`:

- физически удаляются существующие каталоги с устаревшими Superpowers plans/specs в `brand-size-chart`, `browser-vpn-runtime`, `workflow-control-center`, `marketplace-tr-priority` и `workflow-container-tools`;
- `marketplace-tr-priority/.gitignore` начинает игнорировать удалённый каталог;
- из `marketplace-tr-priority/AGENTS.md` удаляются требования, завязанные на Superpowers lifecycle и устаревшее расположение документации;
- остальные несвязанные инструкции и исторические git commits сохраняются;
- после установки и проверки обновлённого `workflow-container-tools` локальный plugin Superpowers удаляется из Codex.

Новый skill и его reference не упоминают удалённую систему или её каталоги.

## Проверка и Acceptance Criteria

- Plugin и skill проходят официальные structural validators.
- Обновлённый plugin устанавливается и показывает `goal-brainstorm` как доступный explicit skill.
- Scenario review подтверждает dedicated-spec режим, direct-owner режим, изменение существующей dated spec, multi-repository coordination, конфликт с `AGENTS.md`, существующую active goal, недоступный status при изменении существующего goal-файла и отдельные fallback-ветки для недоступных status и creation tools.
- Scenario review подтверждает выполнение `Verification Design` из `plugins/workflow-container-tools/skills/goal-brainstorm/references/specification-contract.md` без написания тестового кода во время brainstorm.
- Проверки не утверждают correctness instruction artifacts через поиск конкретных фраз, заголовков или путей.
- Semantic reread подтверждает единственного владельца каждого требования и отсутствие дублирования между instructions, design, spec и goal.
- Ни в одном из пяти перечисленных проектов не остаётся физического устаревшего documentation tree; в Git не остаётся его tracked files.
- `marketplace-tr-priority/AGENTS.md` больше не требует удалённый lifecycle.
- Локальная установка Superpowers отсутствует, а `workflow-container-tools` остаётся установленным и валидным.
- Реализация не добавляет implementation-plan workflow, автоматические commits или скрытую goal activation.
